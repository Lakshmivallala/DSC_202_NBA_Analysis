NBA Game Log Data Structure¶

The dataset follows a Document-per-Game model. Each record captures the performance of a player for a specific calendar date within the 2023-24 season.

Key Data Management Challenges:

  • Sparse Fields: Rows marked as Inactive or Did Not Dress contain no statistical data but serve as placeholders for the team's schedule.
  • Summary Anomalies: The final row of the CSV contains aggregated season totals (e.g., 1956 points), which must be filtered out to avoid skewing per-game averages.
  • Non-Atomic Fields: The MP (Minutes Played) field is stored as a string (MM:SS), requiring transformation for numerical analysis.
  • Schema Flexibility: While most fields are numeric, MongoDB's ability to store these alongside string-based status updates (Inactive) demonstrates why a NoSQL approach is superior to a rigid SQL schema for raw game logs.

3. Advanced Analytics: Server-Side Aggregation Pipeline¶

This cell executes a multi-stage MongoDB Aggregation Pipeline to perform data cleaning and statistical analysis directly on the database engine.

Pipeline Stages:

  1. \$match: Filters for documents containing valid performance metrics.
  2. \$project & \$convert: Dynamically casts string-based statistics (like "PTS") into doubles while handling "Inactive" or "DNP" entries.
  3. Outlier Filtering: Removes "Season Total" rows by filtering for individual game scores within realistic ranges (0-100).
  4. \$group: Calculates the Population Standard Deviation ($stdDevPop) to determine player Consistency, alongside average points and game counts.
  5. \$sort: Ranks players from most reliable (lowest deviation) to most volatile.
In [6]:
# 3. BASIC PERFORMANCE: Reliability Summary

4.2 Server-Side Feature Engineering: Efficiency & Workload¶

While raw totals (Points/Rebounds) provide a surface-level view, this cell utilizes a multi-stage MongoDB Aggregation Pipeline to calculate "Efficiency" metrics that do not exist in the raw CSV data.

Technical "Pro" Features:

  • On-the-Fly Computation: We use $addFields and $divide to calculate Points Per Minute (PPM).
  • Computational Offloading: By performing these calculations on the Database Server rather than in Python, we significantly reduce network latency and memory overhead.
  • Hybrid Metrics: We create a Total_Offensive_Impact score by summing disparate fields (Points + Assists) directly within the pipeline.

Why this proves MongoDB Necessity: In a traditional relational setup, creating these calculated "virtual fields" across a massive dataset often requires complex Views or expensive sub-queries. MongoDB handles this fluidly within the aggregation pipeline, allowing for "Real-Time Analytics" as new data is ingested.

In [8]:
# 4.2 EFFICIENCY: Points Per Minute (PPM)
efficiency_pipeline = [
    # Filter games with significant minutes to avoid outliers
    { "$match": { "MP": { "$gt": 5 } } },
    {
        "$project": {
            "Player": 1,
            "Efficiency_Rating": { "$divide": ["$PTS", "$MP"] },
            "Impact_Score": { "$add": ["$PTS", "$AST"] }

            "_id": "$Player",

            "Total_Impact": { "$sum": "$Impact_Score" }
        }
    },
    { "$sort": { "Avg_PPM": -1 } }
]

4.3 Time-Series Analysis: MongoDB Window Fields¶

To analyze player performance trends over time, we implement the \$setWindowFields stage. This is a high-performance feature that allows us to calculate moving averages without multiple database round-trips.

Technical Execution:

  • Partitioning: Data is partitioned by Player to isolate individual trends.
  • Windowing: A 5-game sliding window (-4, 0) is used to calculate the Rolling Average.
  • Trend Detection: This allows the front office to identify "hot streaks" or cooling periods rather than just looking at season-long static averages.
In [9]:
# 4.3 TIME-SERIES ANALYSIS: 5-Game Rolling Average (Momentum)
# Optimized for your new numeric schema

rolling_avg_pipeline = [
    # Step 1: Filter out DNPs (where PTS is null) and ensure valid dates
    { "$match": { 
        "PTS": { "$ne": None }, 
        "Date": { "$regex": "-" } 
    }},
    
    # Step 2: Sort chronologically for the window to move forward in time
    { "$sort": { "Date": 1 } },
    
    # Step 3: Use Window Fields (The 'Movie' vs 'Snapshot' logic)
    {
        "$setWindowFields": {
            "partitionBy": "$Player",
            "sortBy": { "Date": 1 },
            "output": {
                "Rolling_Avg_PTS": {
                    "$avg": "$PTS",
                    "window": { "documents": [-4, 0] } # Current + 4 previous
                }
            }

5. Advanced Time-Series Analytics: MongoDB Window Fields¶

To identify "Hot Streaks" and performance momentum, we implement the $setWindowFields stage (introduced in MongoDB 5.0).

Technical Justification (Why MongoDB?):

  • Native Time-Series Support: Unlike traditional SQL, which often requires expensive self-joins or complex CTEs (Common Table Expressions) to compare a row with its neighbors, MongoDB's windowing functions operate over a defined "document span" in a single pass.
  • Non-Blocking Performance: By using the documents: [-4, 0] window, the database calculates a 5-game rolling average while the data is still in the pipeline, preventing the need for multiple database round-trips.
  • Contextual Analysis: This allows us to see not just a player's season average, but their momentum leading into a recruitment decision—proving if a player's high scoring is a consistent trend or a short-term fluke.
